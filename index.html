Conceptos

Antes de empezar

Los bloques fundamentales de la IA conversacional

Antes de las demos necesitamos un lenguaje común. Estos 7 conceptos son los que usaremos durante toda la sesión. No te agobies si no los conoces todos — están aquí para que la demo tenga sentido.
CONCEPTO

LLM

Large Language Model. Modelo estadístico entrenado para predecir el siguiente token a partir del texto de entrada. No razona: transforma entradas en salidas basándose en patrones aprendidos del entrenamiento.

"El LLM no sabe quién eres ni qué pasó ayer.
Solo ve el texto que hay en el contexto ahora mismo."
Un LLM es una función texto → texto muy sofisticada. La clave para la audiencia: no tiene memoria entre sesiones, no razona en el sentido clásico, y todo lo que "sabe" está determinado por su entrenamiento + lo que hay en el contexto activo.
CONCEPTO

Modelo

Versión concreta de un LLM con sus propios límites de contexto, velocidad y capacidades: Claude Sonnet 4.5, GPT-4o, Gemini 2.5 Flash… Agent Teams es agnóstico al modelo.

"Mismo AGENTS.md, distinto modelo → comportamiento diferente.
El fichero define el rol; el modelo define la capacidad."
El concepto de modelo es importante para que la audiencia entienda que Agent Teams no depende de un proveedor concreto. El AGENTS.md es portable: el mismo fichero funciona con Claude, GPT-4o o cualquier modelo que soporte instrucciones de sistema.
CONCEPTO

Contexto

La "ventana activa" del LLM: todo el texto que puede procesar en una sesión. Incluye el system prompt, el historial del chat y los archivos adjuntos. Cuando se llena, hay que compactar.

"Si no está en el contexto, para el LLM no existe."
El contexto es el recurso más escaso en un sistema de agentes. Cada token que ocupa el historial es un token menos para razonar sobre el problema actual. Por eso Engram es clave: saca del contexto los datos que no necesitan estar ahí en todo momento.
CONCEPTO

Tokens

La unidad mínima de procesamiento (~4 caracteres en inglés, algo menos en español). Determinan el coste de cada llamada y el límite de la ventana de contexto. Claude Sonnet 4.5 admite hasta 200 000 tokens.

Mensaje corto
~10
tokens
Fichero de código
~1 000
tokens
Codebase grande

tokens
Los tokens son la métrica real que importa: cada API call se cobra por tokens. Un sistema de agentes que reprocesan el mismo contexto una y otra vez es costoso. Engram ayuda a mantener el contexto activo pequeño y relevante.
CONCEPTO

AGENTS.md

El system prompt declarativo del agente. Define su identidad, rol, protocolo y restricciones. Agent Teams lo genera desde un YAML y lo sincroniza en .github/agents/. Cambiar el fichero = cambiar el comportamiento del agente.

.github/agents/{agentId}.md
AGENTS.md es el fichero que Copilot Chat lee como system prompt cuando el usuario invoca @agent-name. Agent Teams lo genera automáticamente desde el AgentSpec YAML y lo mantiene sincronizado. La audiencia no tiene que escribir AGENTS.md a mano.
CONCEPTO

SKILL.md

Instrucciones especializadas inyectadas bajo demanda. No forman parte del system prompt base — se cargan cuando el agente necesita una capacidad concreta: tests, diagramas, APIs...

"Ahorra tokens: solo cargas las instrucciones que necesitas,
cuando las necesitas."
Los skills son bloques de instrucciones reutilizables que el agente puede importar dinámicamente. En lugar de tener un AGENTS.md gigante con todo, tienes uno ligero que carga los skills relevantes para cada tarea. El Skills Browser en el Dashboard muestra todos los skills disponibles.
CONCEPTO

Compactación

Cuando el contexto supera su límite, el LLM (o el IDE) resume el historial en un fragmento más corto. Se pierde detalle. Si los datos críticos solo vivían en el chat, se pierden para siempre.

"Engram es la solución a la compactación.
Los resultados importantes viven fuera del contexto, en disco."
Este es el "aha moment" que conecta conceptos con Engram. La compactación es el enemigo de los sistemas de agentes de larga duración. Engram resuelve esto almacenando los resultados intermedios fuera del contexto. Aunque el chat se compacte, los datos siguen ahí.

☕ Preguntas

¿Alguna duda sobre los conceptos?

~5 min

Pausa breve antes de entrar en arquitectura y demos. Si alguien no tiene claro algún término, ahora es el momento — los usaremos constantemente en el resto de la sesión.

El problema

  • Cada agente reinventa el contexto desde cero
  • Sin coordinación, el trabajo se duplica o se contradice
  • La memoria desaparece al cerrar el chat
  • Imposible paralelizar sin un bus de estado compartido
El pain point clásico: usas Claude/Copilot para tareas grandes, el contexto se pierde, dos agentes hacen lo mismo, no hay orquestación. Agent Teams resuelve esto con arquitectura en capas y memoria persistente (Engram).

¿Qué es Agent Teams?

  • 🎯 Coordinación declarativa — defines equipos en YAML, no en código
  • 🧠 Memoria persistente con Engram — los agentes no repiten trabajo ya hecho
  • ⚡ Paralelismo real — múltiples agentes trabajando simultáneamente sin conflictos
Tres beneficios clave. La coordinación declarativa significa que cambiar de equipo es cambiar un fichero YAML. Engram es el bus de memoria compartida. El paralelismo con aggregator resuelve conflictos automáticamente.

El Router analiza el prompt y decide a quién delegar — nunca ejecuta él mismo. El Orchestrator descompone la tarea en pasos — nunca escribe ficheros. El Worker ejecuta la tarea concreta. Engram es el estado compartido: todos escriben, todos leen. Sin Engram, no hay paralelismo.
"Hoy no voy a explicar código.
Voy a ejecutar prompts y que los agentes trabajen."

Demo 1

Pipeline Simple

Router → Orchestrator → Worker

~20 min · equipo: team-simple
Primera demo. El flujo más básico: un prompt, una tarea, el pipeline completo funcionando de cabo a rabo. Asegúrate de tener team-simple activo en el Dashboard antes de ejecutar.

El prompt

Crea un componente Button con variantes
primary y secondary

→ Ejecutar en @router

Prompt intencionalmente simple. Queremos ver la cadena completa, no la complejidad del resultado. Antes de ejecutar, explica: el router va a analizar este texto, detectar el intent y decidir a qué orchestrator delegarlo.

Lo que va a pasar

Usuario
→
Router
→
Frontend Orchestrator
→
Component Worker
Muestra este slide ANTES de ejecutar el prompt. Explica cada nodo en orden. Después ejecuta el prompt y haz referencia a este flujo durante la ejecución.
ROUTER

El Router analiza el intent

  • 1.Lee el prompt → detecta component_creation
  • 2.Busca en Engram patrones previos de routing
  • 3.Escribe en Engram la evaluación: handoff:{taskId}
  • 4.Llama a agent-teams-handoff → abre chat con el Orchestrator
El Router NUNCA ejecuta código ni escribe ficheros. Solo analiza y delega. Escribe en Engram para que el Orchestrator pueda reconstruir el contexto sin necesitar el historial del chat.
ORCHESTRATOR

El Orchestrator descompone

  • 1.Recuerda el contexto de Engram: handoff:{taskId}
  • 2.Descompone la tarea en pasos atómicos
  • 3.Step 1 → analizar el contexto del proyecto
  • 4.Step 2 → crear el componente Button
  • 5.Delega cada paso al Worker
El Orchestrator NUNCA escribe ficheros ni ejecuta comandos. Solo planifica. La clave: empieza leyendo Engram, no el chat. Esto permite que cualquier agente retome el trabajo en cualquier momento.
WORKER

El Worker ejecuta

  • 1.Recibe las instrucciones del Orchestrator
  • 2.Analiza el contexto del proyecto (estilos, framework…)
  • 3.Genera el componente Button.tsx
  • 4.Persiste el resultado en Engram: frontend:patterns
El Worker es el único que toca ficheros. Escribe en Engram sus patrones para que el siguiente worker en la cadena no tenga que redescubrir el proyecto.
"Cada capa tiene una sola responsabilidad.
El Router no ejecuta. El Orchestrator no escribe.
El Worker no decide."
Este es el principio de diseño central. La separación de responsabilidades es lo que permite el paralelismo de la siguiente demo.
Demo 2

Dispatch Paralelo

Frontend + Backend simultáneamente

~25 min · equipo: team-parallel
Ahora una tarea que cruza fronteras: frontend Y backend a la vez. Aquí es donde la arquitectura en capas brilla de verdad. Cambia al equipo team-parallel en el Dashboard antes de ejecutar.

El prompt

Añade un endpoint de login y el formulario
de login en el frontend

→ Ejecutar en @router

Un solo prompt. El Router detectará que hay dos dominios: frontend Y backend. Nadie le dijo explícitamente "hay frontend y backend". Lo infiere del prompt.

El Router detecta dominios múltiples

Usuario
→
Router
detecta: frontend + backend → llama a agent-teams-dispatch-parallel
Frontend Orchestrator
Backend Orchestrator
El Router escribe en Engram la evaluación para CADA orchestrator por separado. Luego llama a dispatch-parallel con la lista de subtareas. Esto abre DOS chats en paralelo de forma simultánea.

Dos chats en paralelo

🖥 Frontend Orchestrator
  • → Analiza el proyecto React
  • → Diseña el formulario
  • → Delega a Component Worker
  • → Escribe task:{id}:subtask:frontend:result
⚙️ Backend Orchestrator
  • → Analiza el framework API
  • → Diseña el endpoint
  • → Delega a API Worker
  • → Escribe task:{id}:subtask:backend:result
Ambos chats se abren a la vez. Mientras el frontend trabaja, el backend trabaja. Ninguno sabe del otro — solo Engram los conecta. TaskCoordinator monitoriza los ficheros de coordinación y detecta cuándo los dos terminan.

Los Workers ejecutan

Frontend Orch.
→
Component Worker
Backend Orch.
→
API Worker

Cuando ambos terminan → TaskCoordinator dispara al Aggregator automáticamente.

Cuatro agentes trabajando: 2 orchestrators + 2 workers. Todo en paralelo. Cuando los dos orchestrators llaman a complete_subtask, el TaskCoordinator dispara automáticamente al Aggregator.
AGGREGATOR

El Aggregator unifica

  • 1.Recoge task:{id}:subtask:*:result de Engram
  • 2.Detecta conflictos de ficheros entre frontend y backend
  • 3.Produce la respuesta unificada
  • 4.Escribe task:{id}:result
El Aggregator no tiene acceso al chat de los orchestrators. Todo lo que sabe viene de Engram. Aquí es evidente por qué Engram es indispensable.
"Sin Engram, el paralelismo es imposible.
Cada agente trabaja en su propio contexto aislado.
Engram es el bus que los conecta."
Momento de pausa reflexiva. Esto es lo que separa Agent Teams de "usar varios chats de Claude".

Demo 3

Worker Autónomo

Sin Router ni Orchestrator

~15 min · equipo: team-autonomous
No siempre necesitas el pipeline completo. Para tareas bien definidas y acotadas, puedes ir directamente al Worker. Esto es el modo autónomo. ⚠️ Asegúrate de que Engram NO tiene la clave ui:analysis antes de ejecutar.

El prompt

Analiza todos los componentes de /src/ui
y dime cuáles no tienen tests

→ Ejecutar directamente en @ui-analyst-worker

⚠️ Engram debe estar limpio (sin ui:analysis:*)

Remove-Item .engram\engram.db -ErrorAction SilentlyContinue

La tarea es clara y acotada. No necesita descomposición. ¡Anota el tiempo que tarda! Lo vas a usar en la Demo 4. El contraste demo3 vs demo4 es el momento más impactante de la sesión.

Flujo directo

Usuario
→
UI Analyst Worker

Sin indirecciones.
El Worker recoge el contexto de Engram y ejecuta directamente.

La simplicidad aquí es intencional. Para tareas repetibles y acotadas, el overhead del pipeline completo no aporta valor. El Worker autónomo es la versión "script" del sistema.

engram.mode: autonomous

identity:
  id: ui-analyst-worker
  role: worker

engram:
  mode: autonomous         # acepta dispatch directo
  recall:
    - "ui:analysis:{date}" # lee análisis previo si existe
  remember:
    - "ui:analysis:{date}" # persiste el resultado
Dos campos clave: mode: autonomous permite que el Router lo invoque directamente. recall: le dice qué claves leer al empezar. remember: qué escribir al terminar. Esto es lo que habilita la Demo 4.

Patrones de clave en Engram

task : {taskId} : subtask : {agentId}
task : {taskId} : subtask : {agentId} : result
ui : analysis : {date}
frontend : patterns

Cada capa tiene su propio espacio de nombres en Engram.

El naming convention no es arbitrario. El Aggregator busca exactamente task:{taskId}:subtask:*:result para saber qué recoger. Si el worker no escribe con esa clave, el Aggregator no lo encuentra.

Demo 5

Agent Designer

Diseña un agente nuevo desde lenguaje natural

~15 min · sin equipo previo
Agent Designer es el agente que genera AgentSpec YAML a partir de una descripción en lenguaje natural. Lo invocarás con @agent-designer desde Copilot Chat. Esta demo muestra el meta-nivel: usar el sistema para extender el propio sistema.

¿Qué hace Agent Designer?

  • 1.Recibe una descripción en lenguaje natural del agente que necesitas
  • 2.Lee el contexto del proyecto y la arquitectura existente
  • 3.Genera un AgentSpec YAML válido y coherente con el equipo
  • 4.El YAML se puede importar desde el Dashboard directamente

Resultado → un fichero .agent.yml listo para sincronizar

Agent Designer no ejecuta código ni crea ficheros directamente. Su única responsabilidad es producir un YAML válido que puedas revisar antes de importar. Es el único agente que trabaja sin necesitar un equipo activo previo.

El prompt

Necesito un agente que revise Pull Requests:
analice el diff, detecte code smells y
sugiera mejoras siguiendo las convenciones del proyecto

→ Ejecutar en @agent-designer

Prompt intencionalmente concreto: queremos un PR reviewer. Antes de ejecutar, señala que Agent Designer va a leer los agentes ya existentes en el proyecto para que el nuevo sea coherente con el equipo.

El YAML generado

identity:
  id: pr-reviewer
  role: worker
  name: PR Reviewer

instructions:
  - Analiza el diff del PR buscando code smells
  - Comprueba la coherencia con los patrones en Engram: frontend:patterns
  - Sugiere mejoras respetando las convenciones del proyecto

engram:
  recall:
    - "frontend:patterns"
    - "task:{taskId}"
  remember:
    - "review:{prId}:result"
Señala los tres bloques clave: identity (quién es), instructions (qué hace), engram (qué lee y qué escribe). El agente no necesita saber nada más sobre la plomería del sistema — Agent Teams lo gestiona todo.

"Creas agentes describiendo lo que necesitan hacer,
